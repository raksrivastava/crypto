# E2EE Payload Simulator - UIDAI Sandbox Technical Lead Assignment

A simulated end-to-end-encrypted identity payload pipeline: a React form
encrypts mock data client-side, a Python backend decrypts and publishes it
as a Kafka event, and a from-scratch rate limiter protects the CPU-heavy
decryption endpoint.

All identity data (name, "Aadhaar number", device ID) is synthetic and
generated by the demo form. No real personal data is ever sent or stored.

## Structure

```
backend/    FastAPI service - Prompt 1 (decryption, event publishing) + Prompt 3 (rate limiter)
frontend/   React + TypeScript (Vite) - Prompt 2 (encryption form)
```

## Prompt 1 - Backend decryption & event publishing

`POST /api/v1/sandbox/secure-payload`

- `app/crypto/keys.py` generates a 2048-bit RSA keypair once at process
  startup (never hardcoded in source, even as a mock).
- `app/crypto/decryptor.py` is the sole abstraction over decryption:
  base64-decode, then RSA-OAEP(SHA-256) decrypt, then parse JSON. Each
  failure mode raises a distinct, narrow exception.
- On success, a `transaction_id` (UUID4) is attached and the verified
  payload is published as a `Secure_Payload_Received` event via
  `app/events/kafka_publisher.py` (`MockKafkaPublisher`; swap for a real
  `kafka-python`/`confluent-kafka` producer behind the same
  `EventPublisher` interface).
- Error handling (`app/main.py` exception handlers): invalid base64 or
  wrong key returns `400`; malformed JSON or a decrypted payload that
  doesn't match the identity schema returns `400`; publish failure
  returns `502`; anything unexpected returns `500` with no internals
  leaked to the client (full detail is logged server-side).

## Prompt 2 - Frontend encryption component

`frontend/src/components/SecurePayloadForm.tsx`

- Form fields: Name, Mock Aadhaar Number, Device ID.
- On submit: fetches `GET /api/v1/keys/public`, then encrypts the JSON
  form data with RSA-OAEP via the Web Crypto API, running inside a Web
  Worker (`crypto/encryptWorker.ts` + `crypto/encryptClient.ts`) so the
  main UI thread never blocks, then POSTs the ciphertext.
- `idle` / `loading` / `success` / `error` states are modeled explicitly.
  The submit button disables during the in-flight request and the UI
  surfaces both the transaction ID and any error message returned by the
  API.

## Prompt 3 - Distributed rate limiter

`app/rate_limit/` uses no rate-limiting library.

- `sliding_window.py` implements a sliding-window-counter algorithm from
  scratch: it blends the previous window's request count (weighted by
  how much of it still overlaps "now") with the current window's count.
  This avoids the burst-at-boundary flaw of a fixed window while needing
  only two cache reads per check.
- `app/cache/redis_cache.py` defines the `CacheClient` contract
  (`get` / `set` / `increment`) that a real Redis client would implement.
  `InMemoryRedisCache` is the mock used here, matching the assignment's
  "mock external caches" allowance for a distributed Kubernetes
  deployment.
- `middleware.py` is a Starlette/ASGI interceptor scoped to
  `POST /api/v1/sandbox/secure-payload` only. Once a partner's limit
  (default 10 req/sec) is exceeded it returns `429` immediately;
  decryption never runs.

## Running locally

### Backend

```bash
cd backend
python -m venv .venv
./.venv/Scripts/activate        # Windows; use `source .venv/bin/activate` on macOS/Linux
pip install -r requirements.txt
cp .env.example .env            # optional, defaults already match this file
uvicorn app.main:app --reload --port 8000
```

Health check: `GET http://localhost:8000/health`

### Frontend

```bash
cd frontend
npm install
cp .env.example .env            # optional, VITE_API_BASE_URL defaults to localhost:8000
npm run dev
```

Open `http://localhost:5173`.

## Configuration

All tunables are environment variables (see `backend/.env.example`);
none are hardcoded:

| Variable                      | Default                     | Purpose                           |
| ----------------------------- | --------------------------- | --------------------------------- |
| `RSA_KEY_SIZE`              | `2048`                    | Mock keypair size                 |
| `RATE_LIMIT_MAX_REQUESTS`   | `10`                      | Requests/window per`partner_id` |
| `RATE_LIMIT_WINDOW_SECONDS` | `1`                       | Rate-limit window size            |
| `KAFKA_TOPIC`               | `Secure_Payload_Received` | Event topic name                  |
| `CORS_ALLOWED_ORIGINS`      | `http://localhost:5173`   | Allowed frontend origin(s)        |

## Verification performed

- Backend: exercised the valid-payload path (encrypt with the fetched
  public key, decrypt, publish, `200` with a transaction ID), each error
  path (corrupt base64, wrong-key ciphertext, missing field; all clean
  `400`s), and a 15-request burst against one `partner_id` (first 10
  succeed, the rest return `429`).
- Frontend: `npm run build` passes under strict TypeScript with no
  errors; the golden path (fill form, submit, success message with
  transaction ID) was driven end-to-end in a real headless Chromium
  session with zero browser console errors.
